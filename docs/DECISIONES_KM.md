# Decisiones de arquitectura — Knowledge Management
**Registro condensado (ADRs).** Detalle completo en `KM_PLAN.md`. ADRs #1-17: 2026-06-11. ADR #18: 2026-06-13.

| # | Decisión | Por qué | Descartado |
|---|----------|---------|-----------|
| 1 | **Mem0** como memoria episódica transversal | MIT, ~200MB, API REST, namespacing nativo por `user_id`, embeddings locales sin API keys externas | Hindsight (solo-Paperclip), Zep (CE deprecado 04/2025, exige Neo4j + OpenAI key), Khoj (5GB RAM) |
| 2 | Embeddings **locales** (Ollama + nomic-embed-text, 768d) | $0, 20-50ms en CPU, sin dependencia externa | Embeddings externos (innecesario) |
| 3 | LLM de extracción de Mem0 = **modelo gratuito del lab** | $0, key ya existente en el lab | Proveedor externo de pago (no necesario) |
| 4 | Wrapper **FastAPI propio** para Mem0 (no el server oficial) | el oficial no trae embedder Ollama (rebuild igual) y suma wizard/JWT innecesarios | server oficial de Mem0 |
| 5 | **LLM Wiki plugin** = contexto en Paperclip | el "KB plugin" del plan original NO existe en este build; LLM Wiki (patrón Karpathy file-based) es el equivalente real y lee el knowledge en vivo | esperar un KB plugin que no llega |
| 6 | Wiki root del plugin = **sub-árbol `wiki/` rw anidado** dentro del knowledge ro de cada empresa | el plugin necesita escribir (standups/log/ideas); el curado queda intacto; un solo árbol por empresa, visible en Obsidian | wiki separada del knowledge (fragmentación), root en el curado (escrituras fallan) |
| 7 | pgvector instalado pero **sin consumidor activo** | era prerequisito del KB plugin inexistente; queda como capacidad (la migración modernizó la imagen) | desinstalarlo (costo cero mantenerlo) |
| 8 | Protección de contenedores = **bind address**, no UFW | Docker escribe iptables y bypassea UFW — reglas UFW para puertos Docker dan falsa seguridad | reglas UFW para puertos Docker (no manejaban DOCKER-USER) |
| 9 | Auth de Outline = **Google vía OIDC genérico** | el plugin nativo de Google rechaza Gmail personal; OIDC con mismas credenciales no | Google plugin nativo, Workspace pago, SMTP magic link |
| 10 | **Atribución de sesiones del host → empresa primaria** | CC/OC/Hermes son fuentes del host sin empresa; etiqueta `[company:<id>]` enruta excepciones vía clasificador | repartir por heurística (impreciso), duplicar a todas (contaminación) |
| 11 | Aislamiento multi-empresa en Paperclip = **mounts granulares por empresa** (hoy) → **sandbox providers** (cuando haya datos de clientes) | los agentes comparten contenedor con bash/file tools: el enforcement del plugin es blando; no montar = único aislamiento duro sin sandboxes | montar todo (fuga silenciosa), segunda instancia de Paperclip (~1.2GB/empresa, prematuro) |
| 12 | Reestructura: knowledge y ops a **`~/ai-lab/`** (neutral), empresa primaria solo su negocio; symlinks de compatibilidad | la infra es multi-empresa. Bonus: `ai-lab/knowledge` ya sincronizaba a Obsidian | dejar el naming confuso (deuda creciente) |
| 13 | Monitoreo = **Uptime Kuma solo** | 50MB, healthchecks + dead-man's-switch del cron + Telegram | Netdata (sin valor hoy), Prometheus (overkill), scripts custom (3 sistemas solapados) |
| 14 | Secrets = **`.env` 600 + fuera de git + rotación puntual** | lab de un operador; BWS agrega dependencia y fricción diaria sin reducir el riesgo real | Bitwarden Secrets Manager (reevaluar si entra un segundo operador) |
| 15 | NLM = **sync mensual semi-manual**, nunca automatizado | cookies expiran ~14 días; automatizar con Xvfb+Chromium es frágil | script de oauth-rotation automatizado |
| 16 | **NotebookLM por capa: Hermes/CC/humano SÍ directo; Paperclip NO** | (a) técnicamente imposible bien: el contenedor no tiene uvx/cookies; (b) arquitectónicamente incorrecto: NLM no tiene namespacing — acceso directo expondría todos los cuadernos del operador a agentes de cualquier empresa. El conocimiento de NLM entra a cada empresa VÍA DESTILACIÓN dirigida (notebook_query → doc curado en companies/<id>/). Además: config de opencode DEDICADA para el contenedor (mínimo privilegio) en `~/ai-lab/stacks/paperclip-config/` — los agentes ya no heredan los MCP del operador. |
| 17 | **NLM Gateway: acceso mediado y determinista de Paperclip a cuadernos** (amplía #16) | El caso real lo exige: empresas con años de documentación ya indexada en NLM — reprocesarla en sesiones tomaría semanas. Gateway en el host (FastAPI + nlm CLI): allowlist por empresa en `notebooks.yaml` (los IDs nunca salen del gateway), API key, audit log, rate limit 30/h. + `nlm-distill.sh`: batería de preguntas → siembra el knowledge curado en minutos. Fase 2 planificada: perfil de cuenta Google del lab (`nlm login switch`) para que los cuadernos de empresas ni vivan en la cuenta personal del operador. |
| 18 | **Engram como memoria compartida de proyecto (cross-agente)** | Capa que faltaba: los agentes de desarrollo (Claude Code, OpenCode, Antigravity, Codex, Qwen) no comparten conocimiento entre sesiones ni entre ellos a nivel de proyecto. Engram: binario Go, SQLite+FTS5, MCP stdio (sin servicio residente), dedup por hash, topic_key upserts, detección de conflictos semánticos. NO duplica Mem0 (empresa) ni CLAUDE.md (directivas estáticas) ni NLM (conocimiento profundo). | Memorix (requiere Node.js), ogham-mcp (requiere PostgreSQL), archivos planos .ai-memory/ (no manejan concurrencia), shared-memory-mcp (requiere coordinador) |
